# Helium

XML toolkit for Go covering XML parsing, SAX2-style streaming, XPath 3.1,
XInclude, XSD, Relax NG, and Schematron. Started as a libxml2-style port to
Go and grew broader native Go APIs and features along the way.

## XPath 3.1 — XSD Version

The xpath3 package targets **XSD 1.1 only**. This means `+INF` is a valid lexical form for xs:double and xs:float, and xs:dateTimeStamp is a recognized type. QT3 tests with `dependency type="xsd-version" value="1.0"` are skipped.

## XSD — Version Toggle

The xsd package defaults to **XSD 1.0** and treats 1.1 as **opt-in** via `Compiler.Version(xsd.Version11)` (or a `vc:minVersion="1.1"` hint on the root `<xs:schema>` when no explicit version is set). The resolved version is frozen onto the compiled `Schema` so the `Validator` applies the same semantics. 1.0 stays the default so existing behavior and goldens are unchanged.

Implemented in 1.1 mode so far: the 1.1-only lexical forms (`+INF` for xs:double/xs:float; year `0000` on the date types — both gated in `internal/xsd/value` via a `value.Version` argument; relaxng is pinned to `value.Version10`); the 1.1 built-in datatypes (xs:dateTimeStamp, xs:dayTimeDuration, xs:yearMonthDuration, xs:anyAtomicType, xs:error); **UPA weakening** (`check_upa.go` `entriesOverlap`: in 1.1 an element competing with a wildcard is not a cos-nonambig violation — the element wins. Element-over-wildcard precedence is now ENFORCED at validation for the CHOICE case (`validate_elem.go` `matchChoice`/`tryMatchChoice`, gated on `vc.version == Version11`): a branch that consumes the current child via an element leaf AS ITS FIRST CONSUMING TERM takes precedence over one that would consume it via a wildcard, regardless of declaration order or nesting. The classifier (`particleConsumesViaElement` → `particleFirstConsumerKind`/`groupFirstConsumerKind`) is PATH-AWARE: it respects compositor order, occurrences, and emptiable prefixes (reusing `particleEmptiable`), so a LEADING wildcard inside a sequence (e.g. `sequence(any skip, element a)`) is correctly classified as wildcard-first and does NOT win element precedence even though a later element leaf in the same group also matches — only bounded first-consumer determination, no backtracking. A `skip` wildcard declared before a typed element (directly or as a leading nested term) thus no longer steals the element's child and false-accepts an invalid value. Selection is COMMIT-NO-FALLBACK: once any branch is element-first for the current child the choice MUST use an element-first branch and never falls back to a wildcard branch, even if the chosen element branch then fails structurally (a later required term missing, e.g. `choice(sequence(a:int, b:int), any skip)` with only `<a>`) or by content. The SEQUENCE case is a remaining limitation — sequence matching is position-based/greedy, so a minOccurs=0 wildcard preceding an element can still consume the element's child); **xs:assert on complex types** (`assert.go`: parsed from complexType/restriction/extension AND simpleContent extension/restriction, pre-compiled via xpath3, evaluated after content validation — EBV false → invalid; inherited down the base chain. `$value` is bound to the element's TYPED simple value for a simpleContent complex type (empty sequence for complex content; the element declaration's default/fixed effective value is substituted for an EMPTY element before typing, so `$value` reflects the schema-normalized value). The test runs against an ISOLATED tree: a deep copy of the element rooted in NO document and stripped of comment/PI nodes, so an absolute path `/`/`//` raises XPDY0050 — the test cannot navigate to ancestors/siblings — while in-scope namespaces — INCLUDING an inherited default namespace (prefix "") so `namespace-uri-for-prefix('', .)` still resolves — are re-declared on the copy and PSVI type annotations are carried onto DESCENDANT elements and ALL attributes (but NOT the assertion-tree ROOT element, whose own type is not yet assigned during its assert, so `data(.)` is untyped — matching Saxon/the conformance tests). A single-schema `xpath3.SchemaDeclarations` adapter (`schema_decls.go`, `schemaDecls`, carrying the schema version) is passed so a node typed with a NAMED user simple type atomizes through its builtin base (e.g. `data(@x) instance of xs:integer`) and a user-defined `cast`/`castable` inside the assertion validates with the schema's version (so a 1.1-only lexical form like year `0000` is castable). BOTH the `castable` AND the `cast` paths (`xpath3/eval_types.go`) are namespace-aware for QName/NOTATION-derived user types — `cast as t:QNameDerived` validates via `ValidateCastWithNS` and returns the namespace-RESOLVED QName/NOTATION value carrying the user type (additive: only when a context-free `CastAtomic` fails and SchemaDeclarations are present, so existing xpath3 cast behavior is unchanged). The `(local, ns)` for that schema lookup is derived from the ALREADY-RESOLVED target type (`Q{ns}local`, via `castTargetParts`/`schemaAnnotationParts`), NOT from re-parsing the lexical prefix, so an UNPREFIXED user type resolved through an `xpathDefaultNamespace` is looked up in the right namespace. When the cast SOURCE is an ALREADY-RESOLVED QName value (e.g. `data(@q)` whose prefix is declared only on the INSTANCE node, absent from the assertion's static namespaces), `qnameCastLexical` re-validates using the value's OWN URI (binding the lexical's prefix to that URI in a copy of the static map) so the cast succeeds instead of failing to re-resolve `prefix:local`. QName/NOTATION atomization (`resolveQNameFromNode`) predeclares the `xml` prefix → the XML namespace without a node binding (matching `xsd.resolveLexicalQName`), so an xs:QName value `xml:lang`/`xml:space` atomizes; and a list-typed node whose item type's builtin base is `xs:QName`/`xs:NOTATION` resolves EACH token against the node's in-scope namespaces (`atomizeListToken`, used by `data()`/comparison; carries the user/list-item type and its `ListItemAtomized` builtin base), so `xs:list itemType="xs:QName"` atomizes in an assertion (e.g. `count(data(@qs))`). The assert evaluator also sets `Evaluator.QNameValueNoDefaultNamespace()` so an UNPREFIXED xs:QName/xs:NOTATION VALUE atomizes to NO namespace (XSD value-space semantics — a QName value, unlike a name, does not pick up the default namespace; a prefixed value still resolves); under this opt-in `castToQName` ALSO skips the default-namespace fallback for an unprefixed cast target value (e.g. `'bare' cast as xs:QName`), so an unprefixed QName VALUE has no namespace there too — default xpath3 behavior (flag off) is unchanged. A self-referential cast (`cast`/`castable as t:T` inside t:T's OWN assertion) is guarded by a per-validation `(type, value)` cast stack threaded through the context (`castGuardKey` in `schema_decls.go`): a repeat fails closed (treated as not castable) instead of recursing to a stack overflow. A QName/NOTATION fixed/default value substituted into an EMPTY element resolves its prefix against the DECLARATION's namespace context (`ElementDecl.FixedNS`/new `DefaultNS`, via `effectiveValueNS`), not the instance's, for both validation and `$value`; a materialized QName/NOTATION DEFAULT attribute has its prefix declared on the element (`materializeQNameAttrValue` — whitespace-COLLAPSES the value before extracting the prefix so `default=" p:x "` still binds `p`, resolves a UNION attribute type's ACTIVE member (`fixedUnionActiveMember`) so a `memberTypes="xs:QName xs:string"` default active in its QName member is materialized too, minting a fresh prefix on a conflicting instance binding) so a later xs:assert/IDC atomizes it in the schema value space — gated to 1.1, so XSD 1.0 inserts the default exactly as authored with no namespace-declaration rewrite (byte-identical serialization)); the **xs:assertion simple-type facet** (`assertion_facet.go`: parsed by `parseFacets`, stored on `FacetSet.Assertions`, evaluated at simple-value validation time with `$value` bound to the TYPED atomic value — a sequence for a list type with each item typed by its active member, a union resolved to its active member FIRST in `buildValueSequence` (so a union whose active member is a LIST, e.g. `memberTypes="IntList xs:string"` with value `1 2`, yields the list-item sequence — two xs:int — not one untypedAtomic) (SCHEMA-AWARE: `typedAtomic`/`buildValueSequence` thread `vc.schema` into `fixedUnionActiveMember` so a member whose own assertion needs `castable as t:T` is selectable, matching the real validation path — `fixedValueMatches`/`fixedUnionMatches`/`crossMemberValueEqual` also thread `vc.schema` (`c.schema` at compile time) so fixed/enumeration comparisons resolve the active member in the right value space), a QName/NOTATION lexical resolved against in-scope namespaces into an `xpath3.QNameValue` — built by `buildValueSequence`/`typedAtomic`/`atomicForType`; the context item is ABSENT, so `.`/`position()`/`last()` raise a dynamic error → unsatisfied; assertions ANDed along the restriction chain; same SchemaDeclarations adapter passed. Because `validateValue` can evaluate these assertion facets, the COMPILE-TIME throwaway contexts that reach it — the attribute default/fixed constraint check (`checkAttrUseConstraints`, `link_refs.go`) and the facet value-against-base / enumeration / union-member checks (`check_facets.go`) — are built with `schema: c.schema`, so a schema-aware cast in a facet assertion doesn't fail closed and reject a VALID schema at compile time. The runtime `typeAcceptsValue` (IDC field) and standalone `TypeDef.Validate` remain on the Version10/nil-schema default — a documented Phase-1 gap); `xpathDefaultNamespace` on xs:assert/xs:assertion and the root `<xs:schema>` (`resolveXPathDefaultNS` whitespace-COLLAPSES the raw value — element-local OR schema-level — before the empty check and the sentinel switch, so `" ##targetNamespace "` resolves like the sentinel rather than a URI literal; `##targetNamespace`/`##defaultNamespace`/`##local` resolved, chameleon-include aware; an IMPORTED schema's root value is carried into the import sub-compiler (`compile_imports.go` sets `impC.schemaXPathDefaultNS`) so imported assertions resolve their own default namespace); XSD 1.1 **attribute inheritance** (`finalizeEffectiveAttrs`: a complex type inherits every base attribute use it does not redeclare, computed for ALL 1.1 derived types — `checkRestrictionAttrs` no longer flags a non-redeclared required base attribute as "missing" in 1.1, so the merge is what keeps that inherited requirement enforced rather than silently dropped. The merge is TOPOLOGICAL across BOTH extension AND restriction derivations — base finalized first via memoized, cycle-guarded recursion, then `checkRestrictionAttrs`/`checkExtensionAttrDuplicates` runs against td's OWN declarations and the finalized base, then td inherits — so an extension of a restriction (or a restriction of an extension, at any depth) reads a complete base attribute set regardless of source order. In 1.1 the extension and restriction passes DEFER all attribute work to `finalizeEffectiveAttrs`; XSD 1.0 keeps the original per-pass copy, byte-identical) plus **simpleContent content-type narrowing** (`TypeDef.ContentSimpleType` derived from a nested `<xs:simpleType>` AND/OR the restriction's direct facets — `parseSimpleContentRestrictionType` COMPOSES them, returning a restriction of the inline type carrying the sibling facets when both are present, so neither set is dropped — and RECORDS each synthetic restriction in `typeDefSources` so `checkFacetConsistency` runs facet applicability/value checks on it (an inapplicable sibling facet, e.g. `xs:minInclusive` on an `xs:string` base, is a compile error); the EFFECTIVE content simple type is composed across the WHOLE simpleContent derivation chain by `effectiveContentSimpleType`, which recurses through simpleContent complex types (marked `TypeDef.IsSimpleContent`) and stops at the underlying simpleType/builtin, re-basing a facet-only narrowing on the effective base content type so ancestor and derived facets compose. `validateSimpleContent` validates the text against it whenever any chain step has facets/assertions or a non-string builtin (`simpleContentNeedsValidation`), so a restriction enumeration / `xs:float` narrowing is enforced AND inherited through a further restriction or extension, and a simpleContent EXTENSION of a named faceted simple type still enforces that base type's facets/assertions; `$value` uses the same composed type. A NESTED `<xs:simpleType>` content type is returned as-is (its OWN base chain, not re-based), so `validateSimpleContent` ALSO validates the value against `effectiveContentSimpleType(td.BaseType)` whenever `td.ContentSimpleType.BaseType != td` — a nested simpleType RESTRICTS, it does not REPLACE, the base content type (§3.4.2.2), so an inherited base `maxLength` still rejects an over-long value the nested type alone would accept. FIXED-value comparison narrows a simpleContent type to its content simple type CENTRALLY inside `fixedValueMatches` (gated on `version == Version11`: `td = effectiveContentSimpleType(td)`), so EVERY caller is consistent — the runtime non-empty element fixed check, the attribute fixed checks, AND the COMPILE-TIME content-model restriction check (`restriction_particle.go` NameAndTypeOK, where a base/derived element `fixed` must be value-space equal). A simpleContent fixed value is thus compared in its NARROWED content simple type — e.g. a content type restricted to `xs:QName` accepts a different-prefix/same-URI value by QName value-space equality instead of a lexical comparison against the outer complex type's base chain (a non-simpleContent type passes through unchanged, so attribute/list-item/union-member callers are unaffected; XSD 1.0 keeps the declared-type comparison, byte-identical). XSD 1.0 keeps its historical per-type gating, byte-identical); and **conditional type assignment** (`alternative.go`: `<xs:alternative>` on an element declaration selects the governing type via the first @test that holds against the element, else a testless default, else the declared type; xsi:type still takes precedence; applied at the root and all three per-child match sites); and **open content** (`opencontent.go`: `<xs:openContent>` interleave/suffix — interleave removes children whose names aren't declared and which match the wildcard, then matches the declared model on the rest; suffix matches the declared model then requires every trailing child to match the wildcard; declared-named children always go through the model per weak-wildcard precedence). **Not yet implemented** (planned, gated behind the same toggle): **xs:defaultOpenContent** (schema-level default), local-attribute `targetNamespace` (1.1 attribute-in-namespace), `xs:all` member `maxOccurs>1` and wildcards-in-all, wildcard `notNamespace`/`notQName`, `xs:override`, the general user-declarable `explicitTimezone` facet, and `vc:typeAvailable`/`vc:facetAvailable` conditional pruning. Do NOT assume these work in 1.1 mode yet. (Limitations: xs:alternative does not yet support an inline anonymous type — use a named @type.)

## XSLT 3.0 — Conformance Scope

The xslt3 package targets **Basic XSLT 3.0** conformance (W3C spec Section 27). The spec defines 8 conformance levels; only "Basic XSLT Processor" is required. The remaining 7 are optional features:

| Feature | Status | Notes |
|---------|--------|-------|
| Basic XSLT Processor | **Target** | Core requirement |
| Schema-Awareness | In progress | `xsl:import-schema`, type annotations |
| Serialization | Implemented | xml/html/text output methods |
| Streaming | Implemented | DOM-materialization; XTSE3430 analysis |
| Higher-Order Functions | Implemented | Via xpath3 |
| XPath 3.1 | Implemented | Via xpath3 |
| Dynamic Evaluation | Implemented | `xsl:evaluate` |
| **Compatibility (1.0/2.0)** | **Not planned** | Optional per spec; 122 tests skipped legitimately |

Do NOT implement XSLT 1.0/2.0 backwards compatibility mode. Tests skipped with reason `"unsupported feature: backwards_compatibility"` or `"unsupported spec: XSLT20"` are intentionally out of scope.
